# 从工作流到工具连接：如何规划 AI Agent 的 Plugin、Skill 和 MCP

给 AI Agent 扩展能力时，Plugin、Skill 和 MCP 经常被放在一起讨论，结果却越说越乱。原因不是它们彼此冲突，而是它们解决的问题不在同一层。

**Skill 规定 Agent 应该怎样完成任务，MCP 规定 Agent 怎样连接外部能力，Plugin 则把一类工作所需的能力组合成可安装、可分发的产品包。**

这套划分适合解释当前 Codex 等 Agent 产品中的工程实践，但不是跨平台不变的词典。MCP 有公开协议规范。Skill 已逐渐形成以说明文件和配套资源表达工作流的常见形态。Plugin 则主要是宿主产品定义的封装单位，不同平台可能用它指浏览器扩展、函数集合、应用集成或完整能力包。因此，规划之前必须先查清宿主对这些词的实际定义。

## 先看三者分别回答什么问题

| 概念 | 核心问题 | 主要内容 | 典型使用者 | 是否天然连接外部系统 |
| --- | --- | --- | --- | --- |
| Skill | 这件事应该怎么做？ | 指令、步骤、判断标准、模板、脚本、参考资料 | Agent、流程设计者 | 否 |
| MCP | Agent 如何发现并调用外部能力？ | 客户端—服务器通信、能力协商、Tools、Resources、Prompts | Agent 宿主、工具开发者 | 是，但连接本身不等于获得权限 |
| Plugin | 怎样把一类能力交付给用户？ | 一个或多个 Skill，以及 App、MCP 集成或配置模板 | 最终用户、管理员、能力发布者 | 不一定，纯 Skill Plugin 可以完全不连接外部系统 |

可以把它们放进同一条链路理解：

```text
用户目标
  ↓
Plugin：提供可发现、可安装的业务能力包
  ├─ Skill：规定任务步骤、质量标准和工具选择
  └─ App / MCP：提供数据与动作
        ↓
     文件、数据库、GitHub、Slack、日历或内部服务
```

这个关系不是强制嵌套。Skill 可以单独存在，MCP Server 也可以被宿主直接配置，不必先包装成 Plugin。Plugin 的价值主要出现在“需要把一套能力交给别人稳定复用”时。

## Skill：把隐性经验变成可执行的方法

Skill 最适合承载稳定的做事方法。例如“写一篇证据驱动的技术文章”不只是生成文字，还包括确认受众、选择结构、核对事实、规划配图、检查质量。把这些要求写进 Skill 后，用户不必在每个任务里重新解释。

一个成熟的 Skill 通常包含四类信息：

1. **触发条件**：什么任务应该使用它，什么任务不应该使用。
2. **执行流程**：步骤顺序、分支条件、需要调用的工具。
3. **质量门槛**：怎样算完成，必须检查什么，哪些结果不能声称已经验证。
4. **配套材料**：模板、示例、脚本和领域参考资料。

Skill 的边界也很明确。它可以告诉 Agent “先读取 Issue，再检查代码，最后生成修复建议”，但不会凭空提供 Issue 数据，也不会自动获得修改仓库的权限。它是流程知识，不是身份凭证、网络通道或授权系统。

以下场景优先使用 Skill：

- 团队反复执行同一类任务，但每个人的步骤和输出质量不一致；
- 任务依赖领域规范、检查清单、写作风格或固定交付格式；
- Agent 已有完成任务所需的原生工具，只是不知道团队希望它如何使用；
- 工作流需要随着经验积累持续修订，但不需要新增外部系统连接。

如果需求只是“读取 GitHub Issue”，创建 Skill 往往不够；如果 GitHub 已经能访问，而问题是“按团队规则分级、复现并形成开发简报”，Skill 才是缺失的一层。

## MCP：统一能力接口，而不是统一业务流程

[Model Context Protocol](https://modelcontextprotocol.io/docs/getting-started/intro) 是连接 AI 应用与外部系统的开放标准。它采用 Host—Client—Server 架构：Agent 所在应用是 Host；Host 为每个 Server 建立 Client 连接；Server 对外暴露标准化能力。

按照 MCP 官方架构，Server 主要提供三类原语：

- **Tools**：可执行函数，例如查询数据库、创建工单、发送消息；
- **Resources**：供应用读取并放入上下文的数据，例如文件内容、数据库结构和知识库记录；
- **Prompts**：由用户显式选择的可复用交互模板。

MCP 解决的是发现、描述和调用能力的接口问题。它不规定 Agent 应采用哪套业务流程，也不规定模型如何管理上下文。一个 Jira MCP Server 可以提供 `search_issues` 和 `create_issue`。但“先查重、再复现、最后按严重程度决定是否创建”仍应由 Skill、Agent 逻辑或人工制度规定。

以下情况适合引入 MCP：

- 同一套外部能力需要接入多个支持 MCP 的 Agent 宿主；
- 工具需要动态发现，并用结构化输入输出降低集成成本；
- 数据或动作位于数据库、SaaS、内部 API 等 Agent 原生能力之外；
- 团队希望把服务实现与具体模型供应商解耦。

以下情况通常不必新建 MCP Server：

- Agent 已有可靠的本地文件、Shell 或浏览器工具；
- 只有一个简单、稳定的内部函数调用，而且没有复用需求；
- 问题本质是流程不一致，而不是能力无法访问；
- 接入后带来的部署、认证、监控和版本维护成本高于收益。

MCP 也不是安全边界本身。远程 Server 是第三方或组织内部服务，数据会离开 Agent 宿主并受对方的数据处理政策约束。即使协议支持 OAuth 和能力协商，系统仍需单独实施最小权限、操作确认、审计和凭证管理，还要做好服务端鉴权与输入校验。连接成功只能证明“通道可用”，不能证明“这个动作已获用户授权”。

## Plugin：面向交付的组合，不是又一种调用协议

在当前 Codex 的产品定义中，[Plugin 是针对某类工作流打包的能力](https://help.openai.com/en/articles/20001256-plugins-in-codex/)，可以包含 Skills、Apps 和 App templates。Skill 提供方法，App 连接外部系统；一个 Plugin 可以同时组合多个 Skill 和多个 App，也可以只有 Skill。

例如，一个“客户周报”Plugin 可以包含：

- 一个 Skill：规定客户周报的结构、数据核对顺序和风险表述规则；
- 一个 CRM App：读取客户账户和商机数据；
- 一个 Google Drive App：读取会议纪要并保存最终文档；
- 一个配置模板：帮助管理员为本组织填写服务地址和授权信息。

Plugin 的价值不是让协议更标准，而是让能力更容易被发现、安装、配置、授权和分发。它适合完整交付某个岗位或业务场景，而不适合承载底层互操作标准。

需要特别警惕名称错觉：安装 Plugin 不等于其中所有能力都能使用。若底层 App 尚未启用、用户没有源系统权限，或管理员只允许读取，Plugin 仍不能越过这些限制。反过来，禁用某个 App 也不一定删除 Plugin 中纯指令型的 Skill。

以下情况适合做 Plugin：

- 多个 Skill 和连接能力共同服务于一个清晰的业务目标；
- 能力需要交给团队安装，并由管理员管理角色、应用和权限；
- 希望提供开箱即用的入口，同时保留组织级配置；
- 需要独立版本、发布说明、依赖声明和退役策略。

如果只是给自己的 Agent 增加一份代码审查清单，单独创建 Skill 通常更轻；如果只是让多个客户端访问内部搜索服务，优先实现 MCP Server；等到“搜索服务 + 研究流程 + 报告模板”需要作为整体交付时，再包装成 Plugin。

## 用四类典型场景做选择

### 场景一：统一团队的代码审查方式

仓库、Git diff 和测试命令都已能访问，缺的是团队统一的检查顺序、严重级别定义和输出格式。这里应先做 **Skill**。只有在还要读取远程 PR、CI 或安全扫描平台，而宿主没有相应工具时，才增加 App 或 MCP。

### 场景二：让多个 Agent 查询内部知识库

核心缺口是外部数据访问和统一接口，应先建设 **MCP Server**，暴露只读搜索与文档资源。知识库的检索顺序、引用规则和“不确定时如何回答”可以另做 Skill。若要把整套问答能力分发给客服团队，再组合成 Plugin。

### 场景三：自动准备销售客户简报

任务同时依赖 CRM、邮件、日历和团队固定模板。这不是一个工具，而是一条业务工作流，适合以 **Plugin** 交付：内部用 Skills 编排步骤，用 Apps 或 MCP 连接数据源，并由管理员控制哪些角色可以访问哪些系统。

### 场景四：只在当前仓库批量改文件

如果 Agent 已有文件编辑和 Shell 能力，就直接使用原生工具。不要为了“架构完整”额外搭建 MCP，也不必把一次性提示封装成 Skill。扩展机制应该消除重复成本，而不是制造新的维护对象。

## 一套可执行的规划方法

规划时不要从“该装哪个 Plugin”开始，而要从任务和能力缺口开始。

### 第一步：写清任务合同

至少明确输入、输出、触发方式、成功标准和禁止动作。例如：

> 输入一个 GitHub 仓库和时间范围；读取合并记录与 CI 结果；输出带来源链接的发布风险报告；不得合并 PR、修改标签或发送消息。

这段合同能立即分开流程要求和能力要求：“发布风险报告的结构”属于 Skill，“读取远程 PR 与 CI”属于连接能力，“不得写入”属于权限策略。

### 第二步：列出能力缺口

把需求拆成三列：

| 缺口 | 应落在哪一层 | 示例 |
| --- | --- | --- |
| Agent 不知道正确步骤 | Skill | 先查发布范围，再查失败 CI，最后评估阻断项 |
| Agent 无法取得数据或执行动作 | MCP / App | 读取 PR、查询监控、创建日历事件 |
| 多项能力无法方便地交付给用户 | Plugin | 打包发布检查 Skill、GitHub App 和监控 App |

先检查宿主原生能力，避免重复建设。能用现有只读工具完成，就不要为了统一名词再包一层远程服务。

### 第三步：画出数据与权限边界

对每个外部连接记录：数据从哪里来、发送到哪里、使用谁的身份、读取哪些范围、允许哪些写操作、是否需要逐次确认、日志保留多久、如何撤销授权。

优先上线只读能力。删除、发送、支付、部署、合并和权限变更等高影响动作，应采用更窄的工具定义、显式确认和可审计记录，而不是只在 Skill 里写一句“请谨慎”。文字约束有帮助，但不能替代技术控制。

### 第四步：先验证单层，再组合

推荐顺序是：

1. 用人工提示验证任务本身值得自动化；
2. 把稳定做法提炼成 Skill，并用代表性案例测试；
3. 只为确实缺失的外部能力接入 App 或 MCP；
4. 当多项能力需要团队复用时，再封装为 Plugin；
5. 用低风险账号试运行，确认权限、失败恢复和审计记录后再扩大范围。

这个顺序避免把尚未理解的流程过早固化成复杂集成。

### 第五步：为变更和退役负责

Skill 要有版本和回归样例。MCP Server 要监控可用性、Schema 变化、认证失败和工具调用结果。Plugin 要记录依赖、兼容范围和发布说明。长期未使用的能力应停用。退役时不仅要删除本地配置，还要撤销 OAuth、Token 和服务端安装项。

## 最容易踩的五个坑

**把 Skill 当成权限系统。** Skill 能约束流程，却无法阻止一个权限过大的底层工具被错误调用。真正的限制应在工具、服务端和审批层落实。

**把 MCP 当成 Agent 框架。** MCP 负责交换上下文和调用能力，不负责规划、记忆、反思、任务分解或模型选择。

**把 Plugin 当成行业统一标准。** Plugin 的结构和生命周期由宿主定义。迁移到另一平台前，要重新核对其打包格式、权限模型和分发方式。

**一开始就开放写权限。** 先用只读链路证明价值，再逐项增加写动作；“能创建工单”不意味着“能关闭工单”。

**工具越多越好。** 大量重叠工具会增加模型选择难度、上下文占用、故障面和供应链风险。每项能力都应有清晰所有者、使用场景和退出机制。

## 最后的判断口诀

面对一个 Agent 扩展需求，可以按下面三句话判断：

- 缺的是**方法和标准**，用 Skill；
- 缺的是**外部数据或动作接口**，用 MCP 或宿主提供的 App；
- 缺的是**面向用户的一体化安装与分发**，用 Plugin。

它们可以组合，但不应混为一谈。合理的架构不是把三者全部装上，而是让每一层只解决自己的问题：Skill 保证做法可复用，MCP 保证能力可连接，Plugin 保证方案可交付；身份、授权、审计和风险控制则始终由宿主与外部系统共同承担。

文中的选型顺序、三层模型和治理清单是基于上述产品定义与协议边界形成的工程建议，不是 MCP 规范的强制要求。Plugin 一词缺少跨平台统一定义，应用到 Codex 之外的产品时，应以该宿主的官方文档为准。
