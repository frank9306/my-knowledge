# AI 编程任务的完整闭环：从需求到验证

编码 Agent 能连续修改很多文件，但工作量不能证明任务已经完成。可靠交付需要一条可检查的闭环，让每一步都有输入、边界和证据。

```text
明确目标 → 读取项目 → 制定计划 → 最小修改 → 验证 → Review → 交付
```

下面以“给用户列表增加状态筛选”为例，拆开这条链路。

## 需求里写清目标、边界和验收

一句“增加状态筛选”仍然留有很多猜测空间。更完整的任务可以这样写：

```text
给用户列表增加按状态筛选，只改前端，不调整 API；
支持 active 和 disabled 两个状态，保留“全部”选项；
补充空状态和筛选测试，完成后运行项目规定的验证命令。
```

目标说明要得到什么，非目标限制修改范围，验收条件告诉 Agent 如何证明完成。涉及数据、认证、部署、付费服务或破坏性操作时，还应明确授权边界。

## 修改前先读取项目事实

Agent 进入仓库后需要检查适用的指令文件、相关代码、manifest、锁文件、Git 状态和测试结构。这个步骤用来回答：

- 项目使用什么框架和包管理器；
- 现有筛选组件和状态类型在哪里；
- 工作区是否有用户未提交的修改；
- 应运行哪些验证命令；
- 哪些文件属于生成产物或禁止修改的区域。

如果项目已经有可复用的筛选模式，应遵循现有实现。为了一个小功能引入新状态库或重写列表架构，通常会扩大风险和审查成本。

## 大改动先给出可验证计划

计划不需要复述所有操作，只需把关键步骤和检查方式对应起来：

```text
1. 确认现有状态类型和列表数据流 → 验证：定位单一数据入口
2. 增加筛选控件和过滤逻辑 → 验证：组件测试覆盖三个选项
3. 处理无匹配结果 → 验证：空状态测试通过
4. 运行项目验证入口 → 验证：测试、类型检查和构建通过
```

小改动可以直接执行。涉及架构、数据库、认证、大范围删除或重大依赖升级时，计划应先交给用户确认。

## 修改范围越小，证据越容易建立

实现只触碰完成需求所需的文件。不要顺手重命名附近变量、统一格式或清理既有死代码。这些变化会让 diff 变大，也会把功能风险和重构风险混在一起。

最小修改并不等于少写测试。测试是需求的一部分，用来固定用户能观察到的行为。状态筛选至少应覆盖“全部”、两个状态和无结果场景；具体测试层级遵循项目已有约定。

## 验证要匹配改动风险

验证从最相关的检查开始，再进入项目级入口。一个前端功能可能依次运行：

```text
目标组件测试
→ 类型检查
→ lint
→ 生产构建
```

若项目提供 `pnpm check` 之类的统一命令，优先使用它。CI 应调用相同或等价的入口，避免本地和远程使用两套标准。

验证报告必须区分三种状态：实际运行且通过，实际运行但失败，没有运行。网络、凭据或环境导致检查无法完成时，应说明原因，不能把源码检查写成测试通过。

## Review 检查 diff，不重复实现过程

Review 关注修改可能带来的回归：

- 筛选是否改变原有排序或分页；
- 状态枚举是否和 API 数据一致；
- 空状态是否在初次加载时误显示；
- 测试是否只验证实现细节；
- 是否混入无关文件和调试输出。

高风险改动还要检查安全、数据迁移和回滚方式。Review 发现问题后，回到最小修改和验证环节，直到证据覆盖验收条件。

## 交付说明只报告可核实事实

最终说明应包含：改了什么，运行了哪些命令，结果如何，用户是否还要操作。比如：

```text
新增用户状态筛选和无匹配空状态，补充了三个筛选场景的组件测试。
已运行 pnpm test 和 pnpm build，均通过。无需额外配置。
```

commit、push、部署和发布不属于代码完成后的自动下一步，除非用户已经明确授权。

## 循环何时必须停止

连续尝试需要刹车。相同错误反复出现、多个检查点没有新证据、达到时间或费用预算、缺少凭据以及遇到合并冲突时，应停止并报告当前证据。循环继续消耗资源不会自动让目标更接近完成。
