# Agent Skills 不在多：选择、安装、更新与退役

Skill 把一类任务的工作方式写成可复用指令。它能补充编码 Agent 的流程，但每安装一个 Skill，也会增加触发冲突、上下文占用和供应链风险。选择 Skill 的依据应是真实任务，不是清单长度。

## 核心组合覆盖意图

通用 Skill 适合按用户意图划分，而不是按技术名词无限扩张：

| 意图 | Skill 示例 | 完成条件 |
| --- | --- | --- |
| 需求与方案 | `think` | 目标、非目标、方案和验收明确 |
| 界面设计 | `ui` | 视觉意图、组件约束和验收明确 |
| Bug 定位 | `hunt` | 找到可复现、可验证的根因 |
| 审查与发布 | `check` | 给出 findings 或残余风险 |
| 环境审计 | `health` | 配置和供应链风险可行动 |
| 调研 | `learn`、`read` | 结论能追溯到来源 |
| 文案 | `write` | 原意保留，表达符合目标语气 |

语言、框架、OCR 和发布平台等专业能力，在对应任务出现后再安装。范围过大的 `python` 或 `react` Skill 容易和项目规则重叠，具体语言约定更适合放在项目文档或按路径加载的参考资料中。

## 判断用户级还是项目级

通用 Review、Debug 和 Research 流程通常适合用户级安装。与仓库脚本、内部服务、CI 和团队发布方式紧密绑定的 Skill 应放在项目中，让团队共享并接受版本控制。

项目发布、数据迁移和内部代码生成尤其需要项目级约束，因为它们依赖当前仓库的权限和事实。个人凭据不应随项目提交，相关配置应留在本地并加入忽略规则。

## 安装前检查六件事

安装前可以逐项确认：

- 来源和维护者是否可信；
- 当前解析到哪个 release、tag 或 commit；
- 是否下载脚本、注册 hook、调用外部命令或读取凭据；
- 触发条件是否会和已有 Skill 冲突；
- Agent 原生能力是否已经覆盖同类任务；
- 如何禁用、卸载并撤销相关授权。

全局安装会影响多个仓库，执行命令前应展示来源、完整命令、安装范围和冲突检查结果。用户批准一个 Skill，不等于批准它调用的付费服务、Git 写操作或数据外传。

## 生命周期从一次真实任务开始

**引入**，先用它解决一个明确问题，记录来源版本和安装范围。不要因为排行榜或推荐列表一次装几十个。

**观察**，看它是否稳定触发、节省了哪些步骤、有没有误触发或与项目指令冲突。一次任务成功还不足以证明适合常驻。

**固定**，确认有效后固定 release、tag 或 commit。跟随 `main` 自动更新会把未经审查的新脚本和规则带进本地环境。

**更新**，一次只更新一个来源，阅读 diff，再运行代表性任务。检查新增外部命令、权限、hook 和触发条件。

**退役**，一个治理周期没有使用、被 Agent 原生能力替代或长期冲突时，禁用或删除。卸载后还要检查残留配置和授权。

## 如何判断 Skill 真的有效

主观上“流程更完整”不代表结果更好。可以固定模型、仓库、任务和时间预算，对比原生 Agent 与启用 Skill 后的表现：

- 是否更容易一次满足验收条件；
- 是否减少返工或漏掉的验证；
- 是否增加过多 token、耗时和人工确认；
- 在小任务上会不会流程过重；
- 失败时能否说明卡在哪里。

不同 Skill 解决不同意图，不适合只看一个总分。TDD Skill 应看测试和实现行为，研究 Skill 应看来源质量，UI Skill 应看页面结果和视觉验收。

## 维护一份快照，不维护收藏夹

公开 Skill 清单应记录同步日期、来源和用途，并明确它只是某个时间点的快照。安装方式、Skill 名称和上游状态会变化，读者执行前仍需检查当前仓库。

每月可以查看最近使用时间、版本、冲突和权限。长期积累但从不退役的 Skill，会让 Agent 更难判断该遵循哪套流程。
