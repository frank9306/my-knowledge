# 如何写一份真正有用的 AGENTS.md

`AGENTS.md` 的价值不在篇幅，而在于把仓库中真实、稳定、会影响交付的约束交给编码 Agent。一份有用的文件应该让刚进入项目的 Agent 回答四个问题：项目是什么、哪些地方能改、怎样验证、哪些动作需要授权。

## 全局规则和项目规则分开

全局规则适合保存跨项目长期成立的内容，比如默认语言、沟通方式、最小修改原则、Git 安全边界和验证诚实性。项目规则只描述当前仓库：技术栈、目录结构、构建命令、测试入口、数据格式与发布约定。

可以用一个简单判断来分层：换到完全不同的仓库后，这条规则还成立吗？仍然成立就放全局；依赖当前目录、脚本、业务和 CI 的内容放项目。

机器私有路径、账号、token 和客户信息不属于任何可提交的规则文件。需要引用凭据时，只记录环境变量名或获取方式，不写值。

## 项目契约应包含什么

项目 `AGENTS.md` 通常包含下面几类信息。

### 项目结构

说明源码、测试、文档、配置和生成产物的位置，并指出禁止直接编辑的目录。不要把整个目录树贴进去，只写会影响修改判断的边界。

### 真实命令

安装、开发、测试、lint、构建和预览命令必须能在当前仓库执行。不要保留模板占位符，也不要写计划中尚未接入的命令。包管理器和锁文件应与仓库一致。

```md
## Commands

- Install dependencies with `pnpm install`.
- Run the development server with `pnpm docs:dev`.
- Verify production output with `pnpm docs:build`.
```

### 修改边界

记录会造成兼容性、数据或发布风险的约束，比如：

- 不修改生成目录；
- 不改变公开 API 或持久化格式；
- 不覆盖用户已有改动；
- 新增依赖前需要确认；
- 删除文件、migration、认证修改需要授权。

### 完成定义

“完成”应能从证据判断。代码任务至少需要说明改了什么、运行了什么验证、结果如何、还有什么没验证。项目若有统一的 `check` 命令，应把它写成首选入口；没有时列出最相关的测试和构建命令。

### 发布规则

写清楚版本、CHANGELOG、commit、push 和部署的关系。自动发布项目还应指出触发分支和 CI 文件。发布授权属于用户决定，不应由 Agent 从“代码已经完成”推断出来。

## 路径规则只在需要时拆分

大型仓库可能有前端、后端、基础设施等不同约束。只有当某类规则确实只适用于一个目录时，才在更近的路径放置额外说明。通用规则仍留在根文件中。

拆分后的优先级要容易理解：越接近当前文件的项目规则越具体，但不能用它绕过更高层的安全边界。相同内容不要在根目录和子目录各复制一份。

## 其他工具使用薄适配层

如果 Claude Code 需要 `CLAUDE.md`，可以只写：

```md
@AGENTS.md
```

其他平台也采用同样思路，只维护必要的导入或平台差异。完整复制会产生两个事实源，后续每次更新都要记得同步。

## 常见的无效写法

**规则太抽象**，比如“写高质量代码”“充分测试”。Agent 无法据此判断该运行哪些命令，也不能证明要求已经满足。

**命令没有验证**，从模板复制的 `npm test` 在仓库里可能根本不存在。错误命令比缺少命令更危险，因为它制造了项目已被治理的假象。

**把所有偏好塞进项目**，称呼、语气和跨项目安全规则在每个仓库重复，修改一次就要同步很多份。

**把工具说明当成项目事实**，Skill 自带流程只能给默认建议。当前仓库的 `AGENTS.md` 和用户本次要求应优先。

**文件持续膨胀**，排障记录、临时决定和背景材料不断追加。稳定规则留下，详细知识放到专门文档，通过链接引用。

## 定期检查比一次写完更重要

项目脚本、CI 和目录会变化。每月或每次大版本后可以检查：命令是否还能执行，是否出现重复规则，生成目录有没有变化，发布流程是否仍准确。发现规则和仓库事实冲突时，以代码、配置和可运行命令为依据更新文档。
