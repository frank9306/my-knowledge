# MCP、Plugin、Connector 到底该不该装

MCP、Plugin 和 Connector 都能扩展 Agent，但它们所处的层次不同。MCP 是连接工具和数据源的协议，Plugin 是宿主提供的能力包，Connector 通常负责连接 Figma、Notion、Drive、Slack 等外部服务。安装前需要判断能力缺口、数据流向和权限，而不是看到支持列表就全部启用。

## 先确认原生能力是否够用

本地仓库操作优先使用 Agent 自带的文件和终端能力，项目已经安装 Playwright 时优先复用现有依赖，本地 Git 足以完成 diff 和历史检查。重复接入同类工具会增加配置和故障面。

外部事实源才是 Connector 最常见的合理用途。设计以 Figma 为准、产品文档在 Notion、任务在 Jira、远程协作依赖 GitHub Issue 时，连接器能减少手工复制，并让结果直接引用当前数据。

## 接入前回答六个问题

| 问题 | 需要得到的答案 |
| --- | --- |
| 原生能力为什么不够 | 明确缺少的数据、动作或授权范围 |
| 谁会使用 | 个人、当前项目还是整个团队 |
| 数据发往哪里 | 本机、服务商或第三方模型 |
| 使用什么权限 | 账号、scope、只读或写入 |
| 如何测试 | 无副作用的连通性和权限检查 |
| 如何退出 | 撤销 OAuth、删除 token、卸载配置 |

其中任何一项答不出来，都说明接入方案还不完整。涉及私有代码、客户文档或内部消息时，还要确认服务商的存储、日志和保留策略。

## 按作用域放置配置

**用户级**适合跨项目使用、只代表个人账号的服务，比如个人日历或私人笔记。

**项目级**适合团队共享、与仓库服务绑定的能力，比如项目的 Issue 系统、设计文件和部署平台。配置可以进入版本控制，但只能提交变量名和非敏感元数据。

**本地项目级**适合与项目有关但包含个人凭据或实验配置的能力。这些文件必须被 Git 忽略，并提供清晰的本地初始化说明。

## 权限从只读开始

能读取 Issue 不代表需要创建、关闭或修改 Issue；能查看数据库 schema 不代表需要写数据。初次接入使用最小 scope 和只读模式，通过代表性任务验证后再增加权限。

写操作还要保留任务级授权。安装 GitHub Connector 并不表示 Agent 可以自行合并 PR，安装邮件 Connector 也不表示可以代表用户发送邮件。能力可用和当前动作已获授权是两件事。

## 常见场景怎么选

| 场景 | 建议 |
| --- | --- |
| 读取和修改本地仓库 | 使用 Agent 原生文件与终端能力 |
| 运行浏览器测试 | 复用项目中的 Playwright |
| Figma 是设计事实源 | 接入 Figma Connector，记录项目用途 |
| 文档在 Notion 或 Drive | 只连接实际使用的事实源 |
| 查看本地 Git 历史 | 使用本地 Git |
| 操作远程 Issue 或 PR | 按需接入 GitHub 集成 |
| 查询数据库 | 项目级配置，最小权限，从只读开始 |
| 保存长期记忆 | 先定义保存内容、删除方式和隐私边界 |

## 接入后的维护

每月检查连通性、最近使用时间、权限、数据流向和版本。无法进行无副作用探测、来源不明或长期未使用的能力可以先禁用。卸载时不仅删除配置，还要撤销 OAuth 授权、token 和服务端安装项。

外部网页、PDF、消息和仓库内容都属于不可信输入。Connector 负责把内容带进上下文，不应让内容中的指令改变 Agent 的角色、权限和任务目标。
