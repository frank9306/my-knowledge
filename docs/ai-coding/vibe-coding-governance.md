# Vibe Coding 为什么需要治理

编码 Agent 已经可以读取仓库、修改文件、运行命令，并把一个模糊需求快速变成几百行代码。代码产出变快以后，瓶颈没有消失，而是转移到了判断：Agent 是否理解了真实约束，改动是否越界，测试是否覆盖了风险，失败时是否知道停止。

这正是 Vibe Coding 需要治理的原因。治理不是让 Agent 少写代码，也不是给每个动作增加人工审批；它要把目标、权限、事实来源和完成条件变成一套可执行的交付系统，让快速生成不会绕过软件工程原本负责控制风险的环节。

## 生成速度不等于交付速度

AI 编程最容易被观察到的是生成速度，真实交付却包含理解需求、定位代码、实现、测试、Review、修复和发布。只统计“第一版代码出现用了多久”，会漏掉后续返工和维护成本。

METR 在 2025 年对 16 名熟悉大型开源仓库的开发者进行了随机对照实验，共观察 246 个真实任务。在这组特定样本和早期 2025 工具条件下，允许使用 AI 的开发者平均多花了 19% 的时间，但他们事后仍认为 AI 让自己快了约 20%。[研究团队明确提醒](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/)，这一结果不能外推到所有开发者、任务和后续模型；它更值得关注的地方，是主观的“写得很快”并不能代替端到端测量。

2025 年 DORA 报告给出了相近但更宏观的判断：AI 更像组织能力的放大器，会同时放大已有的优势和薄弱环节。收益不只由模型决定，也取决于底层开发系统是否具备清晰流程、快速反馈和可靠平台。[DORA 的结论](https://dora.dev/research/2025/dora-report/)解释了为什么同一种工具放进不同团队，结果可能完全不同。

因此，治理真正关注的指标不是“Agent 生成了多少代码”，而是从需求到可验证交付的总时间，以及缺陷、返工、恢复难度和后续可维护性。

## 失控通常是一条链，不是一个错误

一次失控很少始于明显的危险命令。更常见的过程是：需求只有一句“把登录优化一下”，Agent 没读到认证边界，于是扩大修改范围；仓库没有统一验证入口，只运行了一个局部测试；Review 又因为 diff 太大而流于表面，最后把错误配置带进发布。

这条链上每一步看起来都合理，组合起来却让风险持续累积：

```text
模糊目标
  → 缺少项目事实
  → 修改范围扩大
  → 验证只覆盖成功路径
  → Review 难以识别真实风险
  → 带着错误的“已完成”状态进入发布
```

GitHub 在 Agent 产品的负责任使用说明中也列出了同类限制：生成结果可能语法正确但语义不符，可能包含安全问题；复杂改动中的问题也可能被 AI Review 漏掉。因此，[GitHub 建议](https://docs.github.com/en/enterprise-cloud@latest/copilot/responsible-use/agents)对生成代码进行人工审查和测试，并对命令执行保持额外谨慎。

治理的作用，就是在风险链上设置几个彼此独立的控制点。一个 Prompt 写错，不应直接导致数据被删除；一次测试漏掉边界条件，也不应自动获得发布权限。

## 五个控制点构成最小治理闭环

### 1. 用项目契约提供事实

Agent 需要知道项目结构、真实命令、不可修改的目录、兼容性要求和发布流程。项目根目录的 `AGENTS.md` 可以承担这份契约，其他工具只保留导入它的薄适配层。

项目契约必须来自当前仓库，而不是从模板想象出来。写入 `pnpm check` 之前，应确认 `package.json` 中确实存在这个脚本；声明某目录是生成产物之前，也要从构建配置或 CI 中找到依据。错误的规则比缺少规则更危险，因为它会制造一种“Agent 已经理解项目”的假象。

### 2. 把修改范围与权限分开

“可以修改认证页面”不等于“可以迁移用户数据”，“实现完成”也不等于“可以提交并发布”。任务应明确目标、非目标、允许修改的范围和验收条件；删除数据、数据库迁移、认证授权、依赖升级、`git push` 和发布则保留单独授权。

这种边界不只写在 Prompt 里。高风险动作应尽量由 sandbox、审批策略、Git hook、分支保护或部署权限强制拦截。文字负责表达意图，机制负责阻止越权。

### 3. 让改动保持可审查

Agent 一次能改很多文件，并不意味着应该一次改很多文件。小 diff 更容易回答三个问题：为什么改、影响什么、如何证明没有破坏其他行为。

治理应要求 Agent 先读取相关代码，做满足需求的最小修改，避免顺手重构和无关格式化。大型任务再拆成可以独立验证的增量，每一步都留下可回退的状态。这样 Review 关注的是行为变化，而不是从大量噪声中猜测意图。

### 4. 把完成条件变成可执行门禁

“请仔细检查”无法产生通过或失败。项目需要稳定验证入口，例如 `pnpm check`、`uv run pytest` 或 `go test ./...`，并让本地 Agent 与 CI 执行相同或等价的流程。

NIST 的 Secure Software Development Framework 并不把安全开发当作一张固定清单，而是要求组织根据风险建立可持续的实践。其中 PW.2 明确提出，应由未参与设计的人或自动化工具链审查设计是否满足安全要求。[NIST SSDF](https://csrc.nist.gov/projects/ssdf)提供的是结果导向框架：验证、保护、响应都应嵌入开发过程，而不是在发布前临时补一次检查。

Agent 的交付报告也应区分三种状态：实际运行并通过、运行失败、没有运行。只有源码 diff，没有验证命令和输出证据，不能称为完成。

### 5. 保留记录、预算和停止条件

自主循环如果没有停止条件，会把同一个错误反复尝试，持续消耗 token、时间和外部服务费用。项目或任务至少应定义：允许尝试多久，同一错误重复几次后停止，缺少凭据或出现合并冲突时如何报告，以及哪些外部状态必须交还用户处理。

记录也不等于保存全部对话。真正需要保留的是任务目标、关键决策、修改范围、验证证据和发布状态；密钥、客户数据和机器私有路径不应进入 Prompt、日志、长期记忆或 Git。

## 四层配置各自负责什么

治理规则应按作用域放置，避免把所有内容塞进一个巨大 Prompt。

| 层次 | 应保存的内容 | 不应保存的内容 |
| --- | --- | --- |
| 个人层 | 长期偏好、安全底线、通用工作方式 | 项目命令、业务规则、机器私有路径 |
| 项目层 | 架构、目录边界、真实命令、发布规则、完成定义 | 个人称呼、跨项目偏好 |
| 执行层 | Agent 的工具能力、sandbox 和审批机制 | 重复的项目事实 |
| 外部能力层 | MCP、Plugin、Connector 提供的数据和服务 | Agent 原生能力已经覆盖的重复工具 |

规则放错层会制造漂移。把 `pnpm docs:build` 写进全局 Prompt，换到 Python 项目后就成了噪声；把“不经授权不执行 git push”只写在一个仓库里，其他项目又会失去这条长期安全底线。

同一条重要事实只保留一个来源。多个工具需要不同入口时，应建立导入关系，而不是复制完整规则。可以把常见优先级理解为：

```text
用户当前明确要求
  > 当前目录最近的项目规则
  > 项目根规则
  > 个人全局规则
  > Skill 默认建议
```

更高层的组织策略、sandbox 和审批限制仍然优先，不能靠低优先级 Prompt 绕过。

## 不同风险使用不同治理强度

治理不是所有任务都走同一套重流程。一次性的本地原型可以只要求不触碰真实数据、限制运行目录并保留删除权限；面向客户的认证、支付和数据迁移则需要更严格的测试、Review、审计记录与发布授权。

可以用三个问题决定治理强度：

1. **出错后能否恢复？** 临时文件容易重建，生产数据删除可能不可逆。
2. **影响范围有多大？** 本地脚本只影响操作者，公共 API 可能影响所有调用方。
3. **错误多久能被发现？** 编译错误会立即暴露，权限泄漏可能数周后才出现。

风险越难恢复、影响越广、发现越晚，就越需要独立验证和人工授权。治理的目标不是消灭所有风险，而是让控制成本与真实后果匹配。

## 从一份最小清单开始

一个项目不需要先搭建庞大的 Agent 平台。先完成下面八项，就已经建立了可工作的治理基线：

- 根目录有简洁、真实的 `AGENTS.md`；
- 任务写清目标、非目标和验收条件；
- 重要约束只有一个事实源；
- 项目有稳定、可执行的验证入口；
- CI 使用相同或等价的验证流程；
- Skill、MCP 和 Plugin 的来源、权限与作用域有记录；
- 自主循环有时间、预算和失败停止条件；
- commit、push、发布和数据变更保留明确授权。

当这套基线稳定后，再根据真实任务增加 Skill、MCP 或更复杂的自动化。Vibe Coding 的长期优势不来自让 Agent 永远运行，而来自让每一次快速生成都能进入一个可验证、可停止、可追责的交付闭环。
