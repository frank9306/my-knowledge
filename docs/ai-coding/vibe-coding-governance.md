# Vibe Coding 为什么需要治理

编码 Agent 已经能读仓库、修改文件、运行命令，甚至连续工作很长时间。项目失控时，问题往往不在代码生成能力，而在配置层：全局 Prompt 混入项目命令，多份规则互相复制，Skill 和 MCP 越装越多，最后却没有稳定的验证入口。

治理要解决的是一件具体的事：让 Agent 知道目标、边界、事实来源和完成条件，并让关键约束可以被工具验证。

## 四个层次各管一件事

一套可维护的 AI 编程环境可以分成四层：

| 层次 | 保存什么 | 不该保存什么 |
| --- | --- | --- |
| 个人层 | 长期偏好、安全底线、通用工作方式 | 项目命令、业务规则、机器私有路径 |
| 项目层 | 架构、目录边界、真实命令、发布规则、完成定义 | 个人称呼、跨项目偏好 |
| 执行层 | Codex、Claude Code、OpenCode 的工具能力与审批机制 | 重复的项目事实 |
| 外部能力层 | MCP、Plugin、Connector 提供的数据和服务 | Agent 原生能力已经覆盖的重复工具 |

规则放错层会带来漂移。比如把 `pnpm docs:build` 写进全局 Prompt，换到 Python 项目后就成了噪声；把“不经授权不执行 git push”只写在某个仓库中，其他项目又会失去这条安全底线。

## 项目只保留一个事实源

项目根目录的 `AGENTS.md` 适合承担项目契约。它记录目录结构、真实命令、修改边界、验证方式和发布流程。其他工具需要自己的入口文件时，尽量只导入它，不复制完整规则。

多份完整配置短期看起来方便，维护一段时间后通常会出现三类问题：一条命令只更新了一处，同一动作在两份文件中的授权要求不同，Agent 不知道该相信哪个版本。薄适配层能把这些问题缩小到一个事实源。

规则优先级也应明确。一般可以按下面的顺序处理：

```text
用户当前明确要求
  > 当前目录最近的项目规则
  > 项目根规则
  > 个人全局规则
  > Skill 默认建议
```

安全限制可能由更高层的 sandbox、审批策略或组织规则强制，不能靠低优先级文本绕过。

## 日常工作围绕闭环组织

Vibe Coding 容易把注意力放在“装哪个工具”上。稳定交付依赖的是一条可重复的工作流：

```text
明确目标 → 读取项目 → 制定计划 → 最小修改 → 验证 → Review → 交付
```

工具只负责执行其中一部分。测试驱动 Skill 可以约束实现顺序，GitHub Connector 可以读取远程 Issue，浏览器工具可以检查页面，但它们都不能替代验收条件和验证证据。

每个项目最好提供一个稳定验证入口，比如 `pnpm check`、`uv run pytest` 或 `go test ./...`。Agent 交付时应报告实际运行过的命令、结果和未覆盖部分。只有源码 diff，没有验证记录，完成状态仍然不可靠。

## Prompt 不能代替强制边界

Prompt 适合描述“何时做什么”，不适合独自承担高风险控制。删除数据、数据库迁移、认证修改、依赖安装、Git 写操作和发布动作，可以由 sandbox、审批、hook 和 CI 继续限制。

同样，质量要求也不能只写成“请仔细检查”。格式检查、类型检查、测试和构建只要能够确定执行，就应成为脚本或 CI 的一部分。文字规则负责告诉 Agent 为什么和何时运行，确定性工具负责给出通过或失败。

## 一套配置何时算可用

可以用下面几项检查项目是否已经接入治理：

- 根目录有简洁且真实的 `AGENTS.md`；
- 重要约束只有一个事实源；
- 项目有稳定、可执行的验证入口；
- CI 使用相同或等价的验证流程；
- Skill、MCP 和 Plugin 的来源与作用域有记录；
- 敏感信息不进入 Prompt、日志和 Git；
- 自主循环有时间、预算和失败停止条件；
- commit、push、发布和数据变更保留明确的授权边界。

治理不要求搭建一套庞大系统。先把事实源和验证入口理清，再根据真实任务补能力，维护成本会低得多。
