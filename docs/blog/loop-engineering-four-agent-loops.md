# Loop Engineering：一张图看懂四种 Agent 循环

最近看到一张总结 Agent Loop 的图。它没有把 Loop Engineering 描述成某个新框架，而是把 Agent 的持续工作拆成四种结构：turn-based、goal-based、time-based 和 proactive。这个分类最有价值的地方，是把一个容易说得很玄的概念落到了两个具体问题上：**什么触发一次运行，以及什么条件允许它停止。**

## Loop Engineering 到底在设计什么

最基础的 Agent 循环并不复杂：模型读取上下文、调用工具、接收结果，再决定继续调用工具还是输出答案。真正需要工程设计的并不是这个 `while` 循环，而是循环外部的控制系统：

- 如何提供足够且干净的上下文；
- 如何定义可以自动检查的完成标准；
- 如何限制轮次、时间、token 和成本；
- 如何处理工具失败、重复执行和长期无进展；
- 如何让独立验证器对结果说“不”。

Prompt Engineering 关注“这一轮对模型说什么”，Loop Engineering 关注“怎样让系统在少量人工介入下持续靠近目标”。人的角色也从逐步发指令，转向设计目标、边界与验证机制。

## Turn-based：人控制每次启动

Turn-based 是最常见的交互方式。用户发送提示词，Agent 收集上下文、执行操作并检查结果，然后返回响应。用户审核结果，再决定下一条提示词是什么。

这种模式适合需求仍在形成的任务，例如探索一个陌生代码库、讨论产品方案，或实现一个需要频繁确认方向的小功能。它的优势是可控，缺点是每轮都需要人继续推动。要减少无效往返，可以把重复的检查步骤写进 skill，让 Agent 在交付前自行运行测试、查看页面或检查日志。

## Goal-based：把完成判断交给系统

Goal-based 循环从一组明确的成功标准开始。Agent 尝试完成任务后，验证器判断目标是否达成；如果没有，就把它送回循环继续工作，直到通过检查或触及轮次、时间和成本上限。

它适合结果可测量、实现路径不确定的任务。例如“让首页 Lighthouse 分数达到 90，最多尝试 5 次”就比“优化一下首页”更适合循环执行。这里最关键的不是多跑几轮，而是把“完成”写成系统能够判定的事实：测试全部通过、性能指标越过阈值、指定文件生成且校验成功，而不是让 Agent 根据主观感觉宣布结束。

## Time-based：让时钟触发工作

Time-based 循环把启动权交给时间。系统每隔一段时间执行固定任务，例如检查 PR、处理失败的 CI、汇总消息或拉取外部数据。

这种模式适合任务内容相对固定、输入随时间变化的工作。设计时需要特别注意执行频率：检查间隔应匹配外部状态的变化速度，否则只是在重复消耗 token。任务还应具备幂等性，同一个时间窗口被重复执行时，不能产生重复评论、重复记录或重复扣费。

## Proactive：事件到来后自动组织工作流

Proactive 循环由事件或计划任务触发，并在没有人实时参与的情况下选择和执行工作流。例如系统定时检查反馈频道，发现 Bug 后自动完成分类、修复、审查和回复。

它不是单纯把定时任务做得更复杂，而是同时交出了触发权和流程编排权。适合 Bug 报告、Issue 分流、依赖升级等持续存在、输入不可预知但处理规则相对稳定的职责。由于自动化程度最高，它也最需要权限边界、成本上限、审计记录和独立审查，不能仅依赖执行 Agent 自己评价自己的结果。

## 四种循环怎么选

选择哪种循环，不取决于“哪一种更高级”，而取决于任务本身：

| 任务特征 | 适合的循环 | 交给系统的部分 |
| --- | --- | --- |
| 需求仍在探索，每次结果都会改变下一步 | Turn-based | 单轮执行与自检 |
| 结果可量化，但实现路径不确定 | Goal-based | 完成判断与重复尝试 |
| 工作内容固定，只是周期性发生 | Time-based | 运行时机 |
| 持续有新事件，需要自动选择处理流程 | Proactive | 触发、编排与完成闭环 |

一个实用原则是从最简单的结构开始。先让 turn-based 工作流拥有可靠的自检，再把清晰的完成标准提升为 goal-based；确认任务可重复、操作可幂等之后，才考虑 time-based 或 proactive。自动化范围扩大之前，验证能力必须先跟上。

## 真正重要的是循环里的“刹车”

循环能持续运行，不代表它能正确完成工作。一个可用的 Agent Loop 至少需要四类刹车：

1. **硬限制**：最大轮次、超时、token 和费用上限。
2. **进展检测**：识别重复调用、相同错误和长时间没有状态变化。
3. **客观验证**：用测试、类型检查、指标或结构化规则证明目标完成。
4. **人工升级路径**：权限不足、目标冲突或风险过高时停止并请求确认。

因此，Loop Engineering 的核心不是让 Agent 永远工作，而是让它在边界内持续尝试，并且只在有证据时结束。好的循环减少的是人的重复操作，不是人的判断价值。
