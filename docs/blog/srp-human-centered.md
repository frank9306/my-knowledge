# “单一职责”是为谁服务的？——SRP 原则的人本视角解读

**之前看到一篇文章，主要说是 SRP原则是相对人来说的，具体文章找不到了，今天就问了下GPT，让它帮我解读下，别说，它的回答还是很不错的，以下内容是GPT生成：**

## 🧠 什么是 SRP？

**SRP 全称是 Single Responsibility Principle，即单一职责原则。**

经典定义：

> 一个类应该仅有一个引起它变化的原因。

翻译成大白话：

> 一个模块、类、函数，只做一件事，且把这一件事做好。

举个例子：

*   写了一个 `ReportPrinter` 类，它又负责**生成报表内容**，又负责**打印报表** —— 这违反 SRP。
*   正确做法应该是拆成：
    *   `ReportGenerator`：专心生成报表
    *   `ReportPrinter`：只管怎么把报表打出来

* * *

## 👀 为什么说“SRP 是相对于人”的？

这句话其实很有水平。因为——

> “一个改变的原因”这个“原因”不是抽象概念，是人会动它，是人对它提出了不同的修改诉求。

### ✅ 举个现实的例子：

假设你有个类 `User`，定义如下：

```

class User:
    def calculate_salary(self): ...
    def send_email(self): ...
    def update_profile(self): ...


```

*   财务部说：“我们改下工资计算规则”
*   市场部说：“我们想优化邮件发送逻辑”
*   用户运营说：“用户资料现在要加身份证校验”

每一个部门、每一种角色都会动这段代码。

SRP 说：不行！你这类有太多\*\*“人”会改它了\*\*，每个人有自己修改的理由 —— 这违反了 SRP。

### 所以说：

> SRP 是在保护你免受“多个人同时动一段代码”的噩梦。

一旦很多职责混在一个模块里，任何一个“人”的需求变更都会拖垮你整个结构。

* * *

## 🧩 和其他设计原则的关系

*   SRP 是 SOLID 原则的第一个，强调**高内聚，低耦合**的开始
*   **不符合 SRP，其他原则都别谈**，什么开闭、里氏替换都站不住脚

* * *

## 📦 应用场景

### ✅ 类 / 模块

*   一个类尽量只对一类人负责，比如：`数据库操作类`不要混进 `UI 渲染`

### ✅ 函数级别

*   函数只处理一件事，比如：
    *   错误写法：
        
        ```
        
        def handle_request():
            connect_db()
            do_business()
            render_html()
        
        
        ```
    *   正确写法：
        
        ```
        
        def handle_request():
            data = service_layer()
            return render_response(data)
        
        
        ```

* * *

## 🧨 常见误解

| 误解  | 正解  |
| --- | --- |
| SRP 是“函数只能干一件事” | 不光是“干一件事”，而是“**只被一个人（角色）修改的理由**” |
| SRP 会让类很多，结构太分散 | 短期看麻烦，长期会让项目**更可维护** |
| SRP 是代码洁癖 | 不，SRP 是团队协作下防止屎山的**护城河** |

* * *

## 🎯 总结一句话

> SRP 是为了让一个模块/类只为一个“人”负责。人的变更欲望越多，职责就越容易扭曲。

所以，说它“相对于人”——说得一点都没错，它本质上是**代码模块对“人类变更行为”的隔离策略**。
